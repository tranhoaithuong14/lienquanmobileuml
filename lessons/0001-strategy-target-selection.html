Lesson 0001 · Strategy Pattern · ~10 phút đọc

Chọn mục tiêu trong combat: NearestEnemy và LowestHP

Hai cách dùng Strategy pattern để giải quyết quy tắc auto-target trong Liên Quân Mobile.

Tại sao bài này quan trọng

Trong Liên Quân Mobile, mỗi tướng đều phải quyết định đánh ai mỗi lần tung đòn. Game thật chỉ có hai quy tắc auto-target toàn cục: gần nhất (nearest) và máu thấp nhất (lowest HP). Hai quy tắc này được mô tả bằng tiếng Việt trong primary source — bạn có thể xem chi tiết ở research note cục bộ.

Câu hỏi đặt ra: nếu viết lại game bằng Java, ta mô hình hóa hai quy tắc đó như thế nào cho gọn, dễ mở rộng, dễ test? Đó chính là lúc Strategy pattern xuất hiện.

Bài toán: chọn mục tiêu

Mỗi tướng trong Liên Quân là một Hero. Mỗi tướng có một bộ target selector — quy tắc auto-target toàn cục được áp dụng khi tướng tự động đánh. Khi tướng A đứng giữa ba tướng địch B, C, D, targetSelector của A sẽ trả về một trong ba người đó, hoặc null nếu không có ai hợp lệ.

Hai quy tắc tồn tại trong game thật: Nearest — đánh kẻ địch lân cận (Tulen passive Lôi điện); Lowest HP — đánh kẻ có lượng máu thấp nhất (Butterfly Ám Sát, Lorion passive). Mỗi quy tắc là một thuật toán khác nhau cho cùng một việc: chọn một mục tiêu từ danh sách.

Nếu dùng if/else hoặc switch trong class Hero, mỗi lần thêm tướng mới ta phải sửa Hero. Đó là dấu hiệu mạnh rằng một pattern khác sẽ tốt hơn.

Strategy pattern — bộ xương 4 thành phần

Strategy pattern (GoF) gồm bốn thành phần. Học thuộc bốn cái tên này trước khi đi tiếp:

  1. Strategy — interface mô tả hành vi có thể thay thế.
  2. ConcreteStrategy — class implement Strategy, cung cấp một cách làm cụ thể.
  3. Context — class giữ reference tới Strategy và ủy quyền công việc cho nó.
  4. Client — code bên ngoài chọn ConcreteStrategy nào và gắn vào Context.

Trong bài toán của ta: TargetSelector là Strategy; NearestEnemy và LowestHP là ConcreteStrategy; Hero là Context; và code khởi tạo tướng (ví dụ new Yena(new LowestHP())) là Client.

Interface Strategy

public interface TargetSelector {
    /** Chọn một mục tiêu từ danh sách địch.
     *  Trả về null nếu không có mục tiêu hợp lệ. */
    Enemy select(Enemy attacker, List<Enemy> enemies);
}

Context — Hero

public class Hero {
    private final String name;
    private final TargetSelector targetSelector;
    private int currentHp;

    public Hero(String name, TargetSelector targetSelector) {
        this.name = name;
        this.targetSelector = targetSelector;
    }

    public void autoAttack(List<Enemy> enemies) {
        Enemy target = targetSelector.select(asEnemy(), enemies);
        if (target != null) {
            System.out.println(name + " đánh " + target.getName());
        }
    }

    private Enemy asEnemy() { /* ép kiểu — bỏ qua ở lesson này */ }
}

Hero không biết TargetSelector đang là loại nào. Nó chỉ gọi select() và dùng kết quả. Đó là toàn bộ ý tưởng của Strategy.

Strategy #1 — NearestEnemy

Quy tắc: chọn địch có khoảng cách Euclidean nhỏ nhất từ attacker.

Trong game thật: Tulen passive Lôi điện gọi là "công kích kẻ địch lân cận (ưu tiên tướng)" — tức là biến thể có filter thêm. Phiên bản NearestEnemy dưới đây là dạng thuần: không filter, không tính tướng/lính.

public class NearestEnemy implements TargetSelector {
    @Override
    public Enemy select(Enemy attacker, List<Enemy> enemies) {
        if (enemies.isEmpty()) return null;

        Enemy nearest = null;
        double minDistance = Double.POSITIVE_INFINITY;

        for (Enemy e : enemies) {
            double d = distance(attacker.getPosition(), e.getPosition());
            if (d < minDistance) {
                minDistance = d;
                nearest = e;
            }
        }
        return nearest;
    }

    private double distance(Position a, Position b) {
        double dx = a.x - b.x;
        double dy = a.y - b.y;
        return Math.sqrt(dx * dx + dy * dy);
    }
}

Đặc điểm: thuật toán chỉ phụ thuộc vị trí. Không quan tâm HP, role, hay trạng thái. Nhanh, đơn giản, dễ test.

Câu hỏi tự kiểm: nếu attacker đứng giữa hai địch ở khoảng cách bằng nhau, strategy này chọn ai? (Gợi ý: xem vòng for, điều kiện d < minDistance dùng < hay <=?)

Strategy #2 — LowestHP

Quy tắc: chọn địch có HP hiện tại thấp nhất.

Trong game thật: Butterfly Ám Sát — "kẻ địch có lượng máu thấp nhất (ưu tiên tướng)". Lorion passive — "kẻ địch ít máu nhất trong phạm vi lân cận". Đây là một trong những strategy phổ biến nhất của Liên Quân, vì nó map trực tiếp với mục đích "kết liễu kẻ yếu".

public class LowestHP implements TargetSelector {
    @Override
    public Enemy select(Enemy attacker, List<Enemy> enemies) {
        if (enemies.isEmpty()) return null;

        Enemy weakest = null;
        int minHp = Integer.MAX_VALUE;

        for (Enemy e : enemies) {
            int hp = e.getCurrentHp();
            if (hp < minHp) {
                minHp = hp;
                weakest = e;
            }
        }
        return weakest;
    }
}

Đặc điểm: thuật toán phụ thuộc state runtime của enemy (HP hiện tại), không phải vị trí. Vì HP thay đổi mỗi tick, hai lần gọi liên tiếp có thể trả về hai mục tiêu khác nhau.

So sánh nhanh với NearestEnemy: cùng một interface TargetSelector, nhưng input quan trọng khác nhau (vị trí vs HP), output logic khác nhau. Hero không cần biết.

Phạm vi của Target Selector — chỗ Strategy dừng lại

Trong Liên Quân thật còn có các cơ chế chọn mục tiêu khác: khóa mục tiêu (target lock), đánh dấu rồi đánh (Keera, Quillen), skillshot direction (Yorn). Nhưng những cơ chế đó không thuộc về TargetSelector — chúng là logic riêng của từng skill, không phải quy tắc auto-target toàn cục.

Đây là một bài học quan trọng về Strategy pattern: không phải cái gì "chọn mục tiêu" cũng nên dùng cùng một Strategy interface. Mỗi skill có thể cần input khác (hướng bắn, mark state, ...) — input đó không vừa interface select(attacker, enemies). Ép nó vào cùng interface sẽ làm interface bị phình ra và mất tính gọn.

Edge cases của 2 strategy hiện có

Danh sách rỗng: cả NearestEnemy và LowestHP đều có if (enemies.isEmpty()) return null; ở đầu hàm. Caller (Hero) phải xử lý null — bỏ qua đòn đánh là hợp lý.

Tie-break (khoảng cách bằng nhau): điều kiện d < minDistance trong NearestEnemy dùng strict less-than. Nếu hai địch cùng khoảng cách, enemy nào xuất hiện trước trong list thắng. Hành vi này ổn định nhưng phụ thuộc thứ tự list — không phải random. Muốn random khi tie, phải sửa implementation.

Tie-break (cùng HP): tương tự, LowestHP với hp < minHp cũng chọn enemy đầu tiên trong list khi HP bằng nhau. Trong game thật HP thường là số lớn nên tie hiếm, nhưng vẫn cần biết để debug hoặc viết test.

Bài học: Strategy pattern giải quyết "thuật toán nào" nhưng không ép "policy tie-break" — đó là design decision của từng ConcreteStrategy, và mỗi implementation có thể chọn khác nhau.

Quiz — kiểm tra hiểu

Trả lời trước khi mở đáp án. Mỗi câu có 4 lựa chọn cùng độ dài để không bị hint qua format.

  1. Câu 1. Class nào giữ reference tới Strategy?

    1. Subject
    2. Context
    3. Adapter
    4. Factory
    Đáp án
    B) Context — đúng vai trò của Context trong GoF: giữ reference, ủy quyền cho Strategy.

  2. Câu 2. LowestHP phụ thuộc thông tin nào của enemy?

    1. Vị trí hiện tại
    2. Máu hiện tại
    3. Tướng hay lính
    4. Role của enemy
    Đáp án
    B) Máu hiện tại — LowestHP đọc getCurrentHp() và so sánh min.

  3. Câu 3. Cả NearestEnemy và LowestHP trả về gì khi enemies.isEmpty()?

    1. Trả null.
    2. Throw error.
    3. Trả rỗng.
    4. Chọn bừa.
    Đáp án
    A) Trả null — cả hai implementation đều có if (enemies.isEmpty()) return null; ở đầu hàm. Caller (Hero) kiểm tra null trước khi đánh.

  4. Câu 4. Khi hai địch cùng khoảng cách từ attacker, NearestEnemy chọn ai?

    1. Enemy đầu list.
    2. Enemy cuối list.
    3. Enemy random list.
    4. Enemy ở giữa list.
    Đáp án
    A) Enemy đầu list — điều kiện d < minDistance (strict less-than) giữ enemy đầu tiên; enemy sau không thắng nếu cùng khoảng cách. Đây là hành vi ổn định nhưng không phải random.

  5. Câu 5. Logic "marked target" của Keera có nên dùng cùng TargetSelector interface không?

    1. Có, cùng interface.
    2. Không, khác interface.
    3. Có, thêm method.
    4. Không, không pattern.
    Đáp án
    B) Không, khác interface — marked target là logic riêng của skill Keera, không phải quy tắc auto-target toàn cục. Ép vào TargetSelector sẽ làm interface bị phình và mất gọn.

  • Nếu chưa hiểu tại sao TargetSelector là interface chứ không phải abstract class — hỏi.
