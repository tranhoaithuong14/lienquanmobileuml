Choosing a target in combat: NearestEnemy and LowestHP

Two ways to use the Strategy pattern to model the auto-target rules in a MOBA.

Why this lesson matters

In any MOBA, every hero has to decide who to hit on every attack. The genre has exactly two global auto-target rules: nearest and lowest HP. These rules are described in primary sources — see the local research note for details.

The question: if we rebuild this system in Java, how do we model those two rules cleanly, extensibly, and testably? That is exactly where the Strategy pattern shows up.

The problem: target selection

Every hero in a MOBA is a Hero. Each hero has a target selector — the global auto-target rule applied when the hero auto-attacks. When hero A stands among three enemy heroes B, C, D, A's targetSelector returns one of the three, or null if none is valid.

Two rules exist in the genre: Nearest — attack the closest enemy (think of any ranged-marksman kiting pattern, e.g. a "Marksman A" archetype that prioritizes whoever is nearest); Lowest HP — attack the enemy with the lowest current HP (think of an "Assassin A" archetype whose whole kit is built around finishing weakened targets, or a "Mage A" passive that triggers on the lowest-HP enemy in range). Each rule is a different algorithm for the same task: pick one target from a list.

If we used if/else or switch inside Hero, every new hero would require editing Hero. That is a strong signal another pattern would serve better.

The Strategy pattern — four components

The Strategy pattern (GoF) has four components. Memorize these four names before going further:

  1. Strategy — an interface describing an interchangeable behavior.
  2. ConcreteStrategy — a class implementing Strategy, providing one specific way to do it.
  3. Context — a class that holds a reference to a Strategy and delegates work to it.
  4. Client — external code that chooses which ConcreteStrategy to wire into the Context.

In our problem: TargetSelector is the Strategy; NearestEnemy and LowestHP are ConcreteStrategies; Hero is the Context; and the hero construction code (e.g. new Hero("Yena", ..., new LowestHP())) is the Client.

The Strategy interface

public interface TargetSelector {
    /** Pick one target from the enemy list.
     *  Returns null when no valid target exists. */
    Enemy select(Enemy attacker, List<Enemy> enemies);
}

The Context — Hero

public class Hero {
    private final String name;
    private final TargetSelector targetSelector;
    private int currentHp;

    public Hero(String name, TargetSelector targetSelector) {
        this.name = name;
        this.targetSelector = targetSelector;
    }

    public void autoAttack(List<Enemy> enemies) {
        Enemy target = targetSelector.select(asEnemy(), enemies);
        if (target != null) {
            System.out.println(name + " attacks " + target.getName());
        }
    }

    private Enemy asEnemy() { /* cast — skipped in this lesson */ }
}

Hero does not know which kind of TargetSelector it has. It just calls select() and uses the result. That is the entire idea of Strategy.

Strategy #1 — NearestEnemy

Rule: pick the enemy with the smallest Euclidean distance from the attacker.

In the genre: many ranged marksman kiting patterns boil down to "attack the closest enemy" (with optional role filtering). The NearestEnemy below is the pure form — no filtering, no hero-vs-minion weighting.

public class NearestEnemy implements TargetSelector {
    @Override
    public Enemy select(Enemy attacker, List<Enemy> enemies) {
        if (enemies.isEmpty()) return null;

        Enemy nearest = null;
        double minDistance = Double.POSITIVE_INFINITY;

        for (Enemy e : enemies) {
            double d = distance(attacker.getPosition(), e.getPosition());
            if (d < minDistance) {
                minDistance = d;
                nearest = e;
            }
        }
        return nearest;
    }

    private double distance(Position a, Position b) {
        double dx = a.x - a.x;
        double dy = a.y - b.y;
        return Math.sqrt(dx * dx + dy * dy);
    }
}

Characteristic: the algorithm depends only on position. It does not care about HP, role, or status. Fast, simple, easy to test.

Self-check question: if the attacker stands between two enemies at equal distance, which one does this strategy pick? (Hint: look at the for loop and the d < minDistance condition — is it < or <=?)

Strategy #2 — LowestHP

Rule: pick the enemy with the lowest current HP.

In the genre: this is one of the most common MOBA patterns, because it maps directly to "execute the weak one". Assassin archetypes use it for finishers; mage passives use it to trigger on the lowest-HP enemy in range; carry heroes with execute-style damage use it as their targeting rule.

public class LowestHP implements TargetSelector {
    @Override
    public Enemy select(Enemy attacker, List<Enemy> enemies) {
        if (enemies.isEmpty()) return null;

        Enemy weakest = null;
        int minHp = Integer.MAX_VALUE;

        for (Enemy e : enemies) {
            int hp = e.getCurrentHp();
            if (hp < minHp) {
                minHp = hp;
                weakest = e;
            }
        }
        return weakest;
    }
}

Characteristic: the algorithm depends on the enemy's runtime state (current HP), not position. Since HP changes every tick, two consecutive calls can return two different targets.

Quick comparison with NearestEnemy: same TargetSelector interface, but the important inputs differ (position vs HP), and the output logic differs. Hero does not need to know.

Scope of Target Selector — where Strategy stops

The genre also has other targeting mechanics: target lock, mark-then-strike, skillshot direction. But those do not belong to TargetSelector — they are per-skill logic, not global auto-target rules.

This is an important lesson about the Strategy pattern: not everything that "picks a target" should share the same Strategy interface. Each skill can need different inputs (cast direction, mark state, …) — inputs that do not fit select(attacker, enemies). Forcing them into the same interface would bloat it and lose the pattern's cleanliness.

Edge cases of the 2 existing strategies

Empty list: both NearestEnemy and LowestHP start with if (enemies.isEmpty()) return null;. The caller (Hero) must handle null — skipping the attack is reasonable.

Tie-break (equal distance): the d < minDistance condition in NearestEnemy uses strict less-than. If two enemies are at equal distance, the one that appears first in the list wins. This is a stable behavior but list-order-dependent — not random. If you want random on tie, the implementation must change.

Tie-break (equal HP): the same applies — LowestHP with hp < minHp also picks the first enemy in the list on equal HP. In real gameplay HP is usually a large number so ties are rare, but knowing the policy matters for debugging and for writing tests.

Lesson: the Strategy pattern answers "which algorithm" but does not impose "which tie-break policy" — that is a design decision of each ConcreteStrategy, and each implementation may choose differently.

Quiz — check your understanding

Answer before opening the answers. Each question has 4 same-length options so the format does not give away the right answer.

  1. Question 1. Which class holds the reference to Strategy?

    1. Subject
    2. Context
    3. Adapter
    4. Factory
    Answer
    B) Context — the Context's role in GoF is to hold the reference and delegate to Strategy.

  2. Question 2. Which enemy information does LowestHP depend on?

    1. Current position
    2. Current HP
    3. Hero or minion flag
    4. Enemy's role
    Answer
    B) Current HP — LowestHP reads getCurrentHp() and compares min.

  3. Question 3. What do both NearestEnemy and LowestHP return when enemies.isEmpty()?

    1. Return null.
    2. Throw an error.
    3. Return empty.
    4. Pick arbitrarily.
    Answer
    A) Return null — both implementations have if (enemies.isEmpty()) return null; at the top. The caller (Hero) checks for null before attacking.

  4. Question 4. When two enemies are at the same distance from the attacker, which one does NearestEnemy pick?

    1. First enemy in list.
    2. Last enemy in list.
    3. Random enemy in list.
    4. Middle enemy in list.
    Answer
    A) First enemy in list — the d < minDistance (strict less-than) keeps the first enemy; a later enemy does not win on equal distance. Stable behavior, but not random.

  5. Question 5. Should a per-ability "marked target" mechanic share the TargetSelector interface?

    1. Yes, same interface.
    2. No, different interface.
    3. Yes, add a method.
    4. No, no pattern at all.
    Answer
    B) No, different interface — a marked-target mechanic is per-skill logic, not a global auto-target rule. Forcing it into TargetSelector would bloat the interface and lose its cleanliness.
