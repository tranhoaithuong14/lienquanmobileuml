Strategy Pattern — Cheatsheet

One-page summary. Print it and stick it on the wall if you need to.

The four components

RoleIn our problemResponsibility
Strategy TargetSelector (interface) Defines the interchangeable behavior
ConcreteStrategy NearestEnemy, LowestHP, MarkedTarget One concrete way to carry out that behavior
Context Hero Holds the reference to Strategy and delegates work to it
Client Construction code such as new Hero("Yena", ..., new LowestHP()) Chooses which ConcreteStrategy to wire into the Context

Java skeleton — copy and adapt

// 1. Strategy
public interface TargetSelector {
    Enemy select(Enemy attacker, List<Enemy> enemies);
}

// 2. ConcreteStrategy
public class NearestEnemy implements TargetSelector {
    @Override public Enemy select(Enemy a, List<Enemy> es) {
        // ...
    }
}

// 3. Context
public class Hero {
    private final TargetSelector selector;
    public Hero(TargetSelector s) { this.selector = s; }
    public void autoAttack(List<Enemy> enemies) {
        Enemy t = selector.select(this, enemies);
        if (t != null) System.out.println("hit " + t);
    }
}

// 4. Client
Hero yena = new Hero(..., new LowestHP());

Golden rules

  • The Context does not know the concrete Strategy — it only holds a reference to the interface.
  • Each ConcreteStrategy is its own class — no switch/if inside the Strategy.
  • A Strategy may return null — the caller decides how to handle it.
  • A Strategy can be changed at runtime via a setter, not only at construction.
  • Don't confuse Strategy with State — Strategy is chosen by the Client; State transitions are made by the Context itself.

Comparison with alternatives

ApproachWhen to useProblem
if/else inside Hero 1–2 cases, no expected extension Adding a case = editing Hero. Violates Open/Closed.
switch on an enum Fixed enum, few cases Adding an enum = editing the switch. Still violates O/C.
Inheritance (LowestHPHero extends Hero) Static hierarchy, few variants Combinatorial explosion once there are N variation axes.
Strategy pattern Many algorithms for the same behavior, frequent extension One more class. Several small files.
